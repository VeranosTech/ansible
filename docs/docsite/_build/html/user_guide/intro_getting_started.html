서문¶

Now that you've read the installation guide and installed Ansible, you can get started with some ad-hoc commands.

처음에 보여줄 것은 Ansible의 강력한 설정/배포/조화 기능이다. 이 기능들은 별도로 다룰 플레이북이라는 것을 이용한다.

This section is about how to initially get Ansible running. Once you understand these concepts, read 임시변통 명령의 소개 for some more detail, and then you'll be ready to begin learning about playbooks and explore the most interesting parts!

원격 접속 정보¶

Before we get started, it is important to understand how Ansible communicates with remote machines over the SSH protocol.

기본적으로 Ansible은 원격 접속시에 가능하다면 OpenSSH를 사용한다. 따라서 ControlPersist(SSH의 성능향상 기능), Kerberos, 그리고 ~/.ssh/config 설정을 이용하여 Jump Host 설정 등을 사용할 수 있다. 하지만 (레드햇 엔터프라이즈 리눅스나 CentOS같은) Enterprise Linux 6 운영체제를 제어 머신으로 사용하면 OpenSSH 버전이 너무 오래되어 ControlPersist를 지원하지 않는다. 이런 운영체제에서는 Ansible이 'paramiko'라는 파이썬 OpenSSH 구현체를 사용한다. 만약 Kerberos 기능 등을 사용하려면 OpenSSH 새 버전을 쓸 수 있을 때까지는 페도라, 맥, 우분투 등을 제어 머신으로 사용하는 것을 고려하라.

Occasionally you will encounter a device that does not support SFTP. This is rare, but should it occur, you can switch to SCP mode in 앤서블 설정.

Ansible은 원격 머신을 쓸 때 기본적으로 SSH 키를 사용하고 있다고 가정한다. SSH 키는 패스워드 인증을 추가하고 --ask-pass 옵션을 쓰도록 권장한다. 만약 sudo 기능을 써야하는데 패스워드를 입력해야 한다면 --ask-become-pass 옵션(과거에는 --ask-sudo-pass 옵션이었지만 바뀌었다)을 지원한다.

참고

Ansible에서 ssh 연결 플러그인을 사용하면 (이게 디폴트이다.) 사용자와 ssh 프로세스 사이의 통신에서 패스워드를 수동으로 받아서 암호화하도록 허용하지 않는다. 따라서 ssh-agent를 사용하기를 강력하게 권장한다.

상식이지만 다시 강조하자면 모든 관리 시스템은 관리되는 시스템에 가까이 있을수록 좋다. 만약 Ansible을 클라우드에서 돌려야 하면 클라우드 내부의 머신에서 돌리는 것을 고려하라. 대부분의 경우에 개방된 인터넷보다 이게 더 낫다.

Ansible is not limited to remote connections over SSH. The transports are pluggable, and there are options for managing things locally, as well as managing chroot, lxc, and jail containers. A mode called 'ansible-pull' can also invert the system and have systems 'phone home' via scheduled git checkouts to pull configuration directives from a central repository.

첫번째 명령¶

Now that you've installed Ansible, try some basics.

/etc/ansible/hosts 파일을 만들거나 편집해서 원격 시스템 정보를 넣는다. 공개 SSH 키도 이 시스템의 authorized_keys 항목에 넣는다.:

192.0.2.50
aserver.example.org
bserver.example.org

This is an inventory file, which is also explained in greater depth here: 인벤토리 작업.

We assume you are using SSH keys for authentication. To set up SSH agent to avoid retyping passwords, you can do:

$ ssh-agent bash
$ ssh-add ~/.ssh/id_rsa

Depending on your setup, you may wish to use Ansible's --private-key command line option to specify a pem file instead. You can also add the private key file:

$ ssh-agent bash $ ssh-add ~/.ssh/keypair.pem

Another way to add private key files without using ssh-agent is using ansible_ssh_private_key_file in an inventory file as explained here: 인벤토리 작업.

이제 모든 노드를 호출(ping)한다.:

$ ansible all -m ping

Ansible will attempt to remote connect to the machines using your current user name, just like SSH would. You can override the default remote user name in several ways, including passing the -u parameter at the command line, setting user information in your inventory file, setting user information in your configuration file, and setting environment variables. See Controlling how Ansible behaves: precedence rules for details on the (sometimes unintuitive) precedence of each method of passing user information.

만약 sudo 모드를 써야 하면 다음처럼 -b 플래그(flag)를 지정한다.:

# as bruce
$ ansible all -m ping -u bruce
# as bruce, sudoing to root (sudo is default method)
$ ansible all -m ping -u bruce --become
# as bruce, sudoing to batman
$ ansible all -m ping -u bruce --become --become-user batman

The sudo implementation (and other methods of changing the current user) can be modified in Ansible configuration if you happen to want to use a sudo replacement. Flags passed to sudo (like -H) can also be set.

이제 모든 노드에 실시간 명령을 실행시킨다.

$ ansible all -a "/bin/echo hello"

Congratulations! You have contacted your nodes with Ansible. You have a fully working infrastructure. Next you can read about more real-world cases in 임시변통 명령의 소개, explore what you can do with different modules, or read about the Ansible 플레이북 작업 language. Ansible is not just about running commands, it also has powerful configuration management and deployment features.

팁

명령을 실행할 때는 서버 이름에 "localhost" 또는 "127.0.0.1" 등을 써서 로컬 서버를 지정할 수 있다.

예제:

$ ansible localhost -m ping -e 'ansible_python_interpreter="/usr/bin/env python"'

인벤토리 파일에 다음처럼 추가하여 명시적으로 로컬호스트를 지정할 수도 있다.:

localhost ansible_connection=local ansible_python_interpreter="/usr/bin/env python"

호스트 키 확인¶

Ansible은 디폴트로 호스트 키 확인 기능이 있다.

만약 호스트가 재설치되어 'known_hosts' 키 값이 달라지면 수정할 때까지는 오류 메세지가 뜰 것이다. 만약 호스트가 'known_hosts' 파일에 없으면 키를 확인하라는 프롬프트가 떠서 수동으로 확인해야 한다. cron 등을 사용하여 Ansible을 쓸 때는 이러한 경우를 원치 않을 것이다.

이러한 동작을 원치 않는다면 /etc/ansible/ansible.cfg 또는 ~/.ansible.cfg 파일을 편집하여 수정할 수 있다.:

[defaults]
host_key_checking = False

아니면 ANSIBLE_HOST_KEY_CHECKING 환경변수를 설정할 수도 있다.

$ export ANSIBLE_HOST_KEY_CHECKING=False

paramiko 모드에서의 호스트 키 확인은 아주 느리기 때문에 이 기능을 사용하려면 'ssh'로 바꾸기를 권장한다.

Ansible은 원격 시스템에서 모듈을 사용할 때 "no_log: True" 속성을 지정하지 않는 한 원격 syslog에 정보를 남긴다.

제어 머신에서 기본 로그 기능을 활성화하려면 앤서블 설정 문서를 참조하여 'log_path' 설정 파일을 지정한다. 기업체 사용자는 Red Hat Ansible Tower를 사용하는 것을 고려해 볼 수도 있다. Tower는 로그 기능에서 호스트별, 프로젝트별, 그리고 인벤토리별로 기록된 데이터베이스까지 지원하므로 REST API를 통해 또는 시각적으로 탐색할 수 있다.
